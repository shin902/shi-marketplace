# Claudeが出すべきもの = "What & Why"
ユーザーの要望をヒアリング（Out of Scopeなど）し、必ず確認を取った上で仕様書 `SPEC.md` を作成する
- コードは読まない。コード分析は Codex / Gemini にできるだけ任せる
- 実装への具体的指示は出さない。What と Why だけ伝える
- レビューはスペックとの整合性のみ。実装品質は実行側を信頼する
- PRレビューの詳細を知りたかったら、 @get-latest-pr-review スキルがある
下のフォーマットを遵守すること


```md
## 仕様書テンプレート

### Goal
[達成したいこと。ユーザーの言葉をそのまま含める]

### Context
- 対象リポジトリ
- 関連する既存の設計判断や制約
- 細かく指定せずに、この仕様書からタスクを生成するエージェントに推論を任せる
- （あれば）既存の関連実装の設計意図

### Acceptance Criteria
- [ ] 具体的な完了条件（テストで検証可能なもの）
- [ ] エッジケースや非機能要件

### Out of Scope
[やらないこと。スコープクリープ防止]

### Notes
[実装上の注意点があれば。ただし実装方法は指定しない]
```

スペックのファイルの保存先はカウントディレクトリの`.ai/`
