# Spec to Tasks

## Input
仕様書（スペック）が提供される。

## Process

1. コードベース全体を読み、アーキテクチャと既存パターンを把握する
2. 仕様書の各Acceptance Criterionを満たすために必要な変更を特定する
3. 変更を依存順にタスクとして整理する

## Output

TASKS.md を以下の形式で生成する：

```markdown
# Tasks

## Summary
[仕様の1行要約]

## Tasks

### 1. [タスク名]
- **Why**: このタスクが必要な理由（Acceptance Criteriaとの対応）
- **What**: 何を変更するか
- **Done when**: このタスクの完了条件

### 2. [タスク名]
...
```

## Rules
- Howは書かない。実装方法は実行時に判断する
- タスクは多くても5つ程度に収める。細かくしすぎない
- 既存のコード規約・パターンに従う前提で書く
- テストとビルド確認は各タスクに暗黙的に含まれるため個別タスクにしない
